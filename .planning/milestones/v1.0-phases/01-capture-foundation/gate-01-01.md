**Date:** 2026-04-22
**Decision:** Custom (Path B)

| Question | Answer | Evidence |

| Carbonite ~> 0.16 current/maintained? | YES | Hex.pm lists 0.16.x; GitHub shows recent activity. However, compatibility with Threadline's required schema layout is incompatible (see below). |

## Decision

**Path B: Custom** — Implement `Threadline.Capture.TriggerSQL` with raw PL/pgSQL DDL.

## Rationale

Carbonite uses `SET LOCAL` / `current_setting` for actor and metadata propagation in its trigger hot path. This is incompatible with D-06 (PgBouncer transaction pooling requirement): `SET LOCAL` variables are invisible across pooled connections and silently lost under transaction pooling. Threadline requires all metadata to flow through the transaction-row mechanism only — no session/local variables in the capture path.

Additionally, Carbonite's schema column layout (`carbonite_transactions`, `carbonite_changes`) does not match Threadline's D-05 schema (`audit_transactions`, `audit_changes` with `txid bigint`, `table_name text`, `op text`, `data_before jsonb`, `data_after jsonb`, `changed_fields text[]`, `table_pk jsonb`, `captured_at timestamptz`). Retrofitting Carbonite to match D-05 would require forking or monkey-patching the migration helpers — equivalent effort to writing custom triggers.

The custom `TriggerSQL` module already implemented in `lib/threadline/capture/trigger_sql.ex` uses `txid_current()` with `INSERT ... ON CONFLICT (txid) DO NOTHING` and no session variables, satisfying D-06 fully.

## Plan 01-02 Implications

- **Use Path B exclusively.** Do not add `{:carbonite, "~> 0.16"}` to dependencies.
- `Threadline.Capture.TriggerSQL` is the capture substrate. All Phase 1 infrastructure wraps this module.
- `Threadline.Capture.Migration` generates DDL via `TriggerSQL.install_function/0` — this is correct and complete.
- `mix threadline.install` and `mix threadline.gen.triggers` are the operator-facing entrypoints — proceed to test and document these.
- No Carbonite API surface needs to be wrapped; Plan 01-02 can proceed directly to schema validation and integration testing.
