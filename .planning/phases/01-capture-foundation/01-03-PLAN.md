## Tasks

### Task 1: Create `.credo.exs` configuration

**What to do:**

Create `.credo.exs` at the project root. Use Credo's default strict configuration with one adjustment: disable `Credo.Check.Readability.ModuleDoc` for test support modules (they don't need `@moduledoc`).

```elixir
%Credo.Config{
  configs: [
    %{
      name: "default",
      files: %{
        included: ["lib/", "test/"],
        excluded: ["test/support/"]
      },
      strict: true,
      checks: [
        # Disable checks that are too noisy for a v0.1 library
        {Credo.Check.Design.TagTODO, false},
        {Credo.Check.Design.TagFIXME, false}
      ]
    }
  ]
}
```

Run `mix verify.credo` and fix all violations. Common violations to expect:
- Missing `@moduledoc` on new modules — add `@moduledoc false` to test support modules, proper `@moduledoc` to library modules
- Line length violations — wrap at 98 characters (Credo default)
- Unused variables — prefix with `_`

**Acceptance criteria:**
- `.credo.exs` exists
- `mix verify.credo` exits 0 with no violations

**Done condition:** `mix verify.credo` passes.

### Task 3: Create the GitHub Actions CI workflow

**What to do:**

Create `.github/workflows/ci.yml`:

```yaml
name: CI

on:
  push:
  pull_request:

jobs:
  verify-format:
    name: Check formatting
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: erlef/setup-beam@v1
        with:
          elixir-version: '1.17'
          otp-version: '27'
      - name: Install dependencies
        run: mix deps.get
      - name: Check formatting
        run: mix verify.format

  verify-credo:
    name: Run Credo (strict)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: erlef/setup-beam@v1
        with:
          elixir-version: '1.17'
          otp-version: '27'
      - name: Install dependencies
        run: mix deps.get
      - name: Run Credo
        run: mix verify.credo

  verify-test:
    name: Run test suite
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: threadline_test
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    env:
      DB_HOST: localhost
      MIX_ENV: test
    steps:
      - uses: actions/checkout@v4
      - uses: erlef/setup-beam@v1
        with:
          elixir-version: '1.17'
          otp-version: '27'
      - name: Install dependencies
        run: mix deps.get
      - name: Compile (warnings as errors)
        run: mix compile --warnings-as-errors
      - name: Run tests
        run: mix verify.test
```

**Critical constraints to verify in the YAML before finalizing:**
- Job `id:` fields are `verify-format`, `verify-credo`, `verify-test` — these must NEVER be renamed (CI-05)
- The `on:` block has `push:` with no `paths:` filter — all three jobs run on every push to main (CI-06)
- `pull_request:` trigger is present
- PostgreSQL 16 service is on the `verify-test` job only
- `DB_HOST: localhost` env var is set on `verify-test` so `config/test.exs` resolves correctly

**Acceptance criteria:**
- `.github/workflows/ci.yml` exists
- All three jobs have correct stable IDs (`verify-format`, `verify-credo`, `verify-test`)
- `on: push:` has no `paths:` filter (CI-06)
- PostgreSQL 16 service defined correctly on `verify-test`
- `DB_HOST: localhost` env var present on `verify-test`
- YAML is valid (no syntax errors)

**Done condition:** Workflow file is valid YAML with all three jobs and correct structure.

### Task 5: Write CONTRIBUTING.md skeleton

**What to do:**

Create `CONTRIBUTING.md` at the project root. Four sections only (D-12, DOC-04):

```markdown
# Contributing to Threadline

## Development Environment

**Requirements:**
- Elixir 1.15+
- OTP 26+
- PostgreSQL 14+ (PostgreSQL 16 recommended)

**Setup:**
1. Clone the repository
2. Install dependencies: `mix deps.get`
3. Create the test database: `createdb threadline_test` (as the `postgres` user or equivalent)
4. Run the test suite to verify setup: `mix verify.test`

## Running Tests

```bash
mix verify.test          # full test suite (requires PostgreSQL)
mix test test/path.exs   # single file
```

Note: the test suite requires a running PostgreSQL instance. Tests that exercise trigger
behavior run against a real database — they are not excluded from `mix test` and must not be.

## Running All CI Checks

```bash
mix ci.all               # format + credo + tests in sequence
mix verify.format        # formatting only
mix verify.credo         # Credo strict mode only
mix verify.test          # tests only
```

All checks must pass before submitting a PR. The CI pipeline runs the same commands.

## Submitting a Pull Request

1. Fork the repository and create a branch from `main`
2. Make your changes
3. Run `mix ci.all` and fix any failures
4. Open a pull request against `main` with a clear description of what changed and why
```

**Acceptance criteria:**
- `CONTRIBUTING.md` exists at the project root
- Four sections present: dev environment, running tests, running CI checks, submitting a PR
- `mix verify.test` and `mix ci.all` are cited verbatim (CI-01 through CI-04 alignment)
- No silent exclusion language — note explicitly that trigger tests are NOT excluded

**Done condition:** `CONTRIBUTING.md` exists with all four sections.
