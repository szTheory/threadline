## Goal

Produce a binary decision: use Carbonite `~> 0.16` as the capture substrate, or implement custom trigger SQL. The decision must be backed by evidence on all four gate questions. No capture infrastructure is built in this plan.

---

## Context

The Carbonite compatibility gate is an explicit blocker in STATE.md and `01-CONTEXT.md` D-01. The specific concern is whether Carbonite's context propagation mechanism uses a transaction-row insert (PgBouncer-safe) or `SET LOCAL` (silently wrong under transaction pooling). This is a schema-level constraint — it cannot be retrofitted after the fact.

---

## Tasks

### Task 1: Fetch and parse Carbonite's Hex package metadata

**What to do:**
1. Fetch the Carbonite package page on hex.pm to confirm the current published version and maintenance status.
2. Check the GitHub repository for: last commit date, open issues with "abandoned" / "maintenance" labels, and whether a `CHANGELOG.md` or release history exists for 0.16.x.
3. Record: current version, last release date, number of open issues, maintainer activity signal.

**Acceptance criteria:**
- Current Carbonite version is documented (exact semver, e.g. `0.16.2`)
- Release date of the most recent version is recorded
- Repository activity within the last 12 months: YES or NO
- Open critical bugs (crash, data loss, schema corruption): list any found, or "none found"

**Done condition:** Gate question "Is Carbonite ~> 0.16 current and maintained?" is answered with evidence.

---

### Task 2: Verify PostgreSQL ≥ 14 compatibility

**What to do:**
1. Read Carbonite's `mix.exs` or README for stated PostgreSQL version requirements.
2. Check Carbonite's CI configuration (GitHub Actions) for the PostgreSQL version(s) tested against.
3. Search Carbonite's issue tracker for any open or closed issues mentioning "PostgreSQL 14", "PG14", "postgres 14", or compatibility regressions.
4. Check Carbonite's migration helpers for any SQL features that require PostgreSQL 15+ (e.g., `MERGE`, `UNIQUE NULLS NOT DISTINCT`, `pg_walinspector`).

**Acceptance criteria:**
- PostgreSQL minimum version requirement documented (from package metadata or README)
- CI test matrix confirms PG 14 or lower is tested: YES or NO
- Any known PG 14 incompatibilities: list or "none found"

**Done condition:** Gate question "Does Carbonite support PostgreSQL ≥ 14?" is answered with evidence.

---

### Task 3: Verify context propagation mechanism — transaction-row vs. SET LOCAL

**What to do:**
1. Read Carbonite's source for how it groups changes from the same DB transaction. Look specifically at:
   - `Carbonite.Migrations` — does it create a `carbonite_transactions` table? What columns?
   - The trigger function SQL — how does it link `carbonite_changes` to a `carbonite_transactions` row?
   - Does it use `txid_current()` to key the transaction row? Or `pg_current_xact_id()`?
2. Search the source for `SET LOCAL`, `SET SESSION`, `current_setting`, `set_config` — if any of these appear in the trigger path for metadata propagation, document them.
3. Confirm whether `carbonite_transactions` has a `txid` (or equivalent) column and whether the trigger does an `INSERT ... ON CONFLICT DO NOTHING` keyed on that txid.

**Acceptance criteria:**
- Mechanism is one of:
  - **Transaction-row:** `carbonite_transactions` has a txid column; trigger upserts it; no `SET LOCAL` in metadata path → **COMPATIBLE with D-06**
  - **SET LOCAL only:** trigger reads `current_setting(...)` for metadata with no transaction-row path → **INCOMPATIBLE**
  - **Hybrid:** documents which parts use which mechanism
- Specific source file and line references for the mechanism found

**Done condition:** Gate question "Does Carbonite use a transaction-row (not SET LOCAL) for metadata?" is answered with evidence.

---

### Task 4: Assess Carbonite API surface for Threadline wrapping

**What to do** (only if Tasks 1–3 indicate Carbonite is viable):
1. Document the `Carbonite.Migrations` public API: function signatures for installing the schema, installing triggers, and any options relevant to Threadline (prefix, column filtering).
2. Note whether Carbonite supports an `audit`-schema prefix or only works in `public`.
3. Identify any Carbonite features that Threadline does NOT need in Phase 1 (e.g., outbox, processed marker) — these can be ignored.
4. Note any Carbonite features that conflict with Threadline's schema design in D-05 (column naming, UUID vs. integer PKs).

**Acceptance criteria:**
- `Carbonite.Migrations` API functions listed with signatures
- Schema prefix support: YES (configurable) or NO (public only)
- Conflicts with D-05 schema: list or "none"

**Done condition:** If Carbonite passes the gate, Plan 01-02 has enough information to wrap Carbonite's API without further research.
