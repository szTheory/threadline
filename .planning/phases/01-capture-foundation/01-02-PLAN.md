## Goal

A working Elixir library with trigger-backed capture functioning. Every INSERT, UPDATE, and DELETE on an audited table produces correct `AuditChange` rows grouped under an `AuditTransaction`. Mix tasks generate migration files. Integration tests prove all three operations are captured correctly against a real PostgreSQL database.

### Task 2: Create the Ecto schemas — AuditTransaction and AuditChange

**What to do:**

Create `lib/threadline/capture/audit_transaction.ex`:

```elixir
defmodule Threadline.Capture.AuditTransaction do
  use Ecto.Schema

  @primary_key {:id, :binary_id, autogenerate: true}
  @foreign_key_type :binary_id

  schema "audit_transactions" do
    field :occurred_at, :utc_datetime_usec
    field :source, :string
    field :meta, :map

    has_many :changes, Threadline.Capture.AuditChange, foreign_key: :transaction_id
  end
end
```

Create `lib/threadline/capture/audit_change.ex`:

```elixir
defmodule Threadline.Capture.AuditChange do
  use Ecto.Schema

  @primary_key {:id, :binary_id, autogenerate: true}
  @foreign_key_type :binary_id

  schema "audit_changes" do
    belongs_to :transaction, Threadline.Capture.AuditTransaction, foreign_key: :transaction_id

    field :table_schema, :string
    field :table_name, :string
    field :table_pk, :map
    field :op, :string
    field :data_after, :map
    field :changed_fields, {:array, :string}
    field :captured_at, :utc_datetime_usec
  end
end
```

Create `lib/threadline.ex` (top-level module, Phase 1 is @moduledoc only):

```elixir
defmodule Threadline do
  @moduledoc """
  Threadline — trigger-backed audit capture for Elixir, Ecto, and PostgreSQL.

  See `Threadline.Capture.AuditTransaction` and `Threadline.Capture.AuditChange`
  for the Phase 1 capture schema.
  """
end
```

**Acceptance criteria:**
- Both schema files exist with correct field definitions matching D-05 exactly
- `op` field maps to `text` (not Ecto enum — plain string, CHECK constraint is in the DB)
- `table_pk` and `data_after` and `meta` use `:map` type (JSONB in PostgreSQL)
- `changed_fields` uses `{:array, :string}`
- No Phase 2 fields (`actor_ref`, `action_id`, `context_ref`) are present
- `mix compile --warnings-as-errors` still passes after adding these files

**Done condition:** Both schema modules compile without warnings; schema fields match D-05 exactly.

### Task 4: Create the trigger generation and `mix threadline.gen.triggers` task

**What to do:**

**If Path A (Carbonite):**

Create `lib/mix/tasks/threadline.gen.triggers.ex` that wraps Carbonite's trigger installation. Read the gate document for the exact Carbonite API. The task must:
- Accept `--tables table1,table2` or positional args
- Guard against audit table names (D-10, CAP-10): if `audit_transactions` or `audit_changes` is in the list, print an error and exit non-zero
- Generate an Ecto migration that calls Carbonite's trigger installation helpers for each specified table
- Print the migration path on success

**If Path B (Custom):**

Create `lib/threadline/capture/trigger_sql.ex` with a function that generates the PL/pgSQL trigger DDL for a given table. The trigger function must:
- Call `txid_current()` to get the current DB transaction ID
- Upsert an `audit_transactions` row keyed on a `txid bigint` column (add this column to D-05 schema — this is the mechanism from D-06)
- Insert into `audit_changes` with: `table_schema` from `TG_TABLE_SCHEMA`, `table_name` from `TG_TABLE_NAME`, `op` as lowercase `TG_OP`, `table_pk` as JSONB of the row PK column
- For INSERT: `data_after = to_jsonb(NEW)`, `changed_fields = NULL` (all fields are "new")
- For UPDATE: `data_after = to_jsonb(NEW)`, `changed_fields` = array of keys where `NEW.key IS DISTINCT FROM OLD.key`
- For DELETE: `data_after = NULL` (D-11), `table_pk` from OLD, `changed_fields = NULL`
- Return `NEW` for INSERT/UPDATE, `OLD` for DELETE

Then create `lib/mix/tasks/threadline.gen.triggers.ex` that uses `TriggerSQL` to generate a migration with the trigger function and `CREATE TRIGGER` statements.

**Shared requirements (both paths):**

- Guard in task: if `audit_transactions` or `audit_changes` appears in the table list, `Mix.raise/1` with a descriptive error (D-10)
- One migration file per `mix threadline.gen.triggers` invocation
- Migration filename: `{timestamp}_threadline_triggers_{tables}.exs` where `{tables}` is underscore-joined table names
- `mix compile --warnings-as-errors` passes

**Acceptance criteria:**
- `mix threadline.gen.triggers --tables users` generates a migration file with trigger DDL or Carbonite wrapper for the `users` table
- `mix threadline.gen.triggers --tables audit_transactions` exits non-zero with a descriptive error
- `mix threadline.gen.triggers --tables audit_changes` exits non-zero with a descriptive error
- Generated trigger correctly handles INSERT, UPDATE, and DELETE (verified in Task 5)

**Done condition:** Task generates correct migration; audit-table guard works.
