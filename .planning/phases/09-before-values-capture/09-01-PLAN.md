## Goal

Deliver **BVAL-01** persistence path: nullable `changed_from jsonb` on `audit_changes`, install template updated, `TriggerSQL` emits INSERTs that populate `changed_from` **only** on UPDATE when the **opt-in** SQL path is selected, **null** for INSERT/DELETE and for default (non-opt-in) installs; `mix threadline.gen.triggers` gains `--store-changed-from` and `--except-columns col1,col2` (exact flag names) defaulting **off** / empty.

### Task 2: Extend `TriggerSQL` — default path writes `changed_from` NULL; opt-in UPDATE builds sparse JSON

**What to do**

1. Change `install_function/0` to `install_function/1` taking a keyword list with defaults `[store_changed_from: false, except_columns: []]`. Keep `def install_function(), do: install_function([])` for backward compatibility **or** delegate from arity-0 to arity-1 per project style — either way **call sites** in `migration.ex` `inspect(TriggerSQL.install_function())` must resolve (update to `inspect(TriggerSQL.install_function([]))` if needed).
2. In the emitted SQL `INSERT INTO audit_changes (...)`, append column **`changed_from`** and value **`NULL::jsonb`** for **INSERT** and **DELETE** branches always.
3. For **UPDATE** when `store_changed_from` is **false**, insert **`NULL::jsonb`** for `changed_from`.
4. For **UPDATE** when `store_changed_from` is **true**, set a PL/pgSQL variable `v_changed_from jsonb` built only from keys that appear in `v_changed_fields` after removing any key in the **except** list (case-sensitive match to PostgreSQL column names as emitted today). Values come from **`to_jsonb(OLD) -> key`** so SQL NULL becomes JSON null. Use the **same** key discovery as `v_changed_fields` (single computation path — D-14) — implement by reusing the existing `jsonb_each` pattern or a follow-up loop, but **one** source of truth for membership.
5. Add `def install_function_for_table(table_name, opts)` **or** equivalent that returns SQL for `CREATE OR REPLACE FUNCTION threadline_capture_changes_#{table_name}()` **only when** `Keyword.get(opts, :store_changed_from, false)` is true, containing the opt-in body; when false, documentation string only — **exact behavior:** plans must pick one shipped approach consistent with `09-RESEARCH.md` (**per-table function** for opt-in tables calling the opt-in body, shared global `threadline_capture_changes()` remains default-off for UPDATE `changed_from`).

   **Concrete target state:** For `store_changed_from: false` (default `install_function`), global function matches semantics above. `Mix.Tasks.Threadline.Gen.Triggers` with `--store-changed-from` emits migration SQL that defines **`threadline_capture_changes_<table>()`** (snake_case table name) and `CREATE TRIGGER` uses `EXECUTE FUNCTION threadline_capture_changes_<table>()` for each listed table on that run; migrations generated **without** the flag keep using `threadline_capture_changes()`.

**Read first**

- `lib/threadline/capture/trigger_sql.ex`
- `.planning/phases/09-before-values-capture/09-RESEARCH.md` (per-table function recommendation)
- `.planning/phases/09-before-values-capture/09-CONTEXT.md` (D-01–D-05, D-14)

**Acceptance criteria**

- `grep -F 'changed_from' lib/threadline/capture/trigger_sql.ex` exits 0 (non-zero lines in SQL strings).
- **No new session coupling:** `rg 'set_config|SET LOCAL' lib/threadline/capture/trigger_sql.ex` prints **no lines** (empty output, exit code 1 for `rg` is OK). The only `current_setting` usage must remain the existing `threadline.actor_ref` read inside the trigger SQL string.
- `MIX_ENV=test mix compile` exits 0.

**Done condition:** All acceptance criteria pass.

### Task 3: `mix threadline.gen.triggers` — flags and emitted per-table function + trigger wiring

**What to do**

1. Extend `OptionParser.parse/2` `strict:` with:
   - `store_changed_from: :boolean` (default **false** when absent)
   - `except_columns: :string` (default `""` — comma-separated column names, trimmed)
2. When `store_changed_from` is **true** for a run, for **each** table in `--tables`, emit `execute` for `DROP FUNCTION IF EXISTS threadline_capture_changes_<table>() CASCADE` in `down` (or non-CASCADE per project safety — choose **IF EXISTS** drop of function before trigger drop order correctly).
3. Emit `up` that creates **per-table** function using `TriggerSQL.install_function_for_table(table, store_changed_from: true, except_columns: parsed_list)` then `create_trigger` variant that references **`threadline_capture_changes_<table>()`** instead of the global name.
4. When `store_changed_from` is **false**, preserve **today’s** generated migration shape (global `threadline_capture_changes()` in trigger lines).
5. Update `@moduledoc` with examples for the new flags.

**Read first**

- `lib/mix/tasks/threadline.gen.triggers.ex`
- `lib/threadline/capture/trigger_sql.ex` (after Task 2)

**Acceptance criteria**

- `grep -F 'store_changed_from' lib/mix/tasks/threadline.gen.triggers.ex` exits 0.
- `grep -F 'except_columns' lib/mix/tasks/threadline.gen.triggers.ex` exits 0.
- `MIX_ENV=test mix test test/threadline/capture/trigger_test.exs` exits 0 (existing triggers still use global function path — tests may need migration of test setup in plan 09-02 if broken; **if this test fails after Task 3**, adjust test helper to install global function only — record deviation in SUMMARY).
