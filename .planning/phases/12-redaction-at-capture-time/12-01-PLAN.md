## Tasks

### Task 1: Add `Threadline.Capture.RedactionPolicy`

**What to do**

1. Create `lib/threadline/capture/redaction_policy.ex` with functions used at **codegen / Mix** time (not runtime DB):
   - `validate!/1` — accepts keyword or map with `:exclude`, `:mask` (lists of String column names), optional `:mask_placeholder` string.
   - If intersection of normalized exclude and mask is non-empty, raise `ArgumentError` whose message contains the literal substring `"exclude"` and `"mask"` and lists at least one offending column name.
   - `validate_placeholder!/1` — reject empty string, length > 200 (constant in module), any `String.contains?(placeholder, <<ch>>)` for `ch` in `0..31`; if invalid, raise with substring `"placeholder"`.
   - Export `default_placeholder/0` returning exactly `"[REDACTED]"`.

**Read first**

- `.planning/phases/12-redaction-at-capture-time/12-CONTEXT.md` (D-05–D-09)
- `lib/threadline/capture/trigger_sql.ex` (module style)

**Acceptance criteria**

- `MIX_ENV=test mix run -e 'Application.ensure_all_started(:threadline); Threadline.Capture.RedactionPolicy.validate!(exclude: ["a"], mask: ["a"])'` exits **non-zero** and stderr or stdout contains both `exclude` and `mask`.

### Task 2: Extend `TriggerSQL` — options API and INSERT/UPDATE `data_after` redaction

**What to do**

1. Extend `install_function/1` to accept a keyword list (backward compatible default `[]`) including:
   - `:exclude`, `:mask` (string column lists, default `[]`)
   - `:mask_placeholder` optional string (default `RedactionPolicy.default_placeholder/0`)
2. When **both** `exclude` and `mask` are empty for the global function, generated SQL must remain **byte-for-byte equivalent** to the current `install_function([])` body for the capture logic (same `CREATE OR REPLACE FUNCTION threadline_capture_changes()` shape) — i.e. no behavioral change for hosts without rules.
3. When either list is non-empty on the global path, inject PL/pgSQL that:
   - Sets `v_data_after` from `to_jsonb(NEW)` on INSERT/UPDATE, then **removes** excluded keys (JSONB `-` operator or equivalent), then **sets** each mask key to `to_jsonb('<placeholder>'::text)` using a **single-quoted literal** embedded in the generated SQL (placeholder validated in Elixir first).
4. UPDATE `changed_fields` computation must **not** list excluded column names; for masked columns, treat as changed when underlying value differs from placeholder vs OLD (follow `12-CONTEXT.md` D-10 — document chosen rule in code comment if subtle).
5. Add `@doc` noting json/jsonb columns use **whole-value** mask (D-12).

**Read first**

- `lib/threadline/capture/trigger_sql.ex`
- `.planning/phases/12-redaction-at-capture-time/12-CONTEXT.md` (D-11, D-12)

**Acceptance criteria**

- `grep -nE 'set_config|\\bSET LOCAL\\b' lib/threadline/capture/trigger_sql.ex` exits **1** (no matches anywhere in the file after changes).

**Done condition:** Compile OK; global function with empty opts unchanged behavior; non-empty opts emit redacted `v_data_after` path.

### Task 3: Per-table function — `install_function_for_table/2` arity compatibility + redaction + `changed_from`

**What to do**

1. Extend `install_function_for_table/2` to accept redaction keys in the opts keyword: `:exclude`, `:mask`, `:mask_placeholder` (in addition to `:store_changed_from`, `:except_columns`).
2. **Backward compatibility:** Existing calls `install_function_for_table(t, store_changed_from: true, except_columns: [...])` must still compile and behave as today when exclude/mask omitted (empty).
