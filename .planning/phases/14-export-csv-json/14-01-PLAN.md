## Threat model

| ID | Threat | Severity | Mitigation in this plan |
|----|--------|----------|-------------------------|
| T-14-01-01 | Unbounded `Repo.all` loads millions of rows into memory; operators mistake partial data for full export | high | Default `max_rows:` (module attribute `@default_max_rows` e.g. `10_000`, document in `@moduledoc`); use `limit: max_rows + 1` probe; return explicit `truncated:` / `returned_count` / `max_rows` in the result map (`14-CONTEXT.md` D-11). Ship `Threadline.Export.stream_page_fun/3` (or `stream_changes/2`) as keyset-paged `Stream` with conservative `page_size` (500–2000) and short `repo.all/2` per step (`D-12`). |
| T-14-01-02 | Unknown filter keys silently ignored → empty or wrong row sets and false confidence | high | Shared `Threadline.Query.validate_timeline_filters!/1` (or equivalent name) raises `ArgumentError` with message listing allowed keys `[:repo, :table, :actor_ref, :from, :to]`; invoked from **both** `timeline/2` and export entrypoints (`D-03`, `14-RESEARCH.md` §6.2). |
| T-14-01-03 | Composable query accidentally double-joins `audit_transactions` (actor filter + export join) producing wrong SQL or duplicate rows | high | Refactor filter pipeline to a **single** `join(:inner, [ac], at in AuditTransaction, on: ac.transaction_id == at.id)` for the shared timeline/export query; `filter_by_actor/2` becomes `where` on existing `at` binding only—**no** second `join` (`D-07`, `14-RESEARCH.md` §6.1). |
| T-14-01-04 | User-controlled strings in filters bypass parameterization | medium | Only `Ecto.Query` composable fragments with `^` pinned parameters (existing `filter_by_*` style); no `String.to_existing_atom/1` on user input for filter **keys** (keys are compile-time validated list). |

**Blocked (must not ship):** any default path that loads unbounded rows without truncation metadata; any export path that skips strict filter validation.

## Goal

Deliver **EXPO-01** and **EXPO-02** in the library: one internal `%Ecto.Query{}` pipeline (`D-02`) shared by `Threadline.Query.timeline/2` and `Threadline.Export`, strict shared filter validation (`D-03`), hybrid CSV (`D-05`) via **NimbleCSV**, versioned JSON wrapper + optional NDJSON (`D-06`, `D-08`), joined transaction context per change without double-join (`D-07`), Jason encoding + semantic test assertions (`D-09`, `D-10`), bounded default + truncation metadata (`D-11`), and a keyset-style stream helper (`D-12`). **No** `Threadline` top-level delegators or Mix task in this plan (14-02).

## Tasks

### Task 1: Dependency + `validate_timeline_filters!/1` + refactor `timeline/2` query pipeline

**What to do**

1. Add `{:nimble_csv, "~> 1.2"}` (or current stable) to `mix.exs` deps; run `mix deps.get`.
2. Add **public** `Threadline.Query.validate_timeline_filters!(filters) :: :ok` — allowed keys exactly `[:repo, :table, :actor_ref, :from, :to]`; duplicate keys may follow `Keyword` normalization; unknown key → `ArgumentError` message **must** contain the substring `allowed` or enumerate allowed keys (grep-verifiable).
3. Add `Threadline.Query.timeline_query(filters) :: Ecto.Query.t()` returning an `AuditChange`-rooted query with:
   - **One** inner join to `Threadline.Capture.AuditTransaction` as `at` on `ac.transaction_id == at.id`.
   - Existing semantics: `filter_by_table` / `filter_by_from` / `filter_by_to` on `ac`; actor filter as `where` on `at.actor_ref` JSON containment (refactor current `filter_by_actor/2` to **not** add a join).
   - `order_by([ac], desc: ac.captured_at)` then **secondary** `order_by([ac], desc: ac.id)` for stable ordering (document; supports keyset tests).
4. `timeline/2`: resolve `repo` exactly as today (`Keyword.get(opts, :repo) || Keyword.fetch!(filters, :repo)`), call `validate_timeline_filters!(filters)`, then `timeline_query(filters) |> select([ac], ac)` so `repo.all` returns `%AuditChange{}` structs with **all** schema fields as today (join stays in SQL for actor filter; Ecto loads root `ac` only).
   - **Critical:** Returned structs must match previous integration test expectations (`AuditChange` fields, same ordering).
5. Export query builder: new private or public `Threadline.Query.export_changes_query(filters)` (name at implementer discretion) that calls same `validate_timeline_filters!`, reuses `timeline_query(filters)`, then extends with `select` of both change and transaction fields needed for export maps (or `preload` **not** used for N+1 avoidance—prefer single `select` with maps).

### Task 2: Implement `Threadline.Export` — CSV + JSON wrapper + NDJSON + truncation + ActorRef maps

**What to do**

1. New module `lib/threadline/export.ex` with `@moduledoc` linking to `Threadline.Query.timeline/2` and stating identical `filters` / `opts` + repo resolution (`D-01`, `D-18` pointer for Hex).
2. **Repo resolution:** same as `timeline/2`: `repo = Keyword.get(opts, :repo) || Keyword.fetch!(filters, :repo)`.
3. **Public functions** (names adjustable but document in plan execution):
   - `to_csv_iodata(filters, opts) :: {:ok, map()}` — map keys include `:data` (`iodata`), `:truncated` (boolean), `:returned_count` (non_neg_integer), `:max_rows` (pos_integer). CSV header row fixed column order: `id`, `transaction_id`, `table_schema`, `table_name`, `op`, `captured_at`, `table_pk`, `data_after`, `changed_fields`, `changed_from`, `transaction_json` (single JSON column for nested `%{id:, occurred_at:, actor_ref:, source:}` built from joined `at`—hybrid `D-05` / `D-07`; **or** split `transaction_*` columns if fewer JSON blobs preferred—**must** document final header list in `@moduledoc`). Datetimes ISO 8601 UTC. JSON columns: `Jason.encode!` into strings; cells written with `NimbleCSV.RFC4180` dump helpers (verify exact function names from installed NimbleCSV docs, e.g. `dump_to_iodata/2`).
   - `to_json_document(filters, opts) :: {:ok, map()}` — `:data` is `iodata` or binary for JSON object: `format_version: 1`, `generated_at`, `changes` (array of change maps). Each change map includes string/atom keys per chosen convention and nested `:transaction` map with at least `id`, `occurred_at`, `actor_ref`, `source` (`D-06`, `D-07`). Serialize `actor_ref` via `Threadline.Semantics.ActorRef.to_map/1` nested inside JSON (`14-RESEARCH.md` §1).
   - Opt `json_format: :wrapped` (default) | `:ndjson` — NDJSON: **no** outer array; one JSON object per line, same per-change map shape as array elements (`D-08`).
4. **Truncation (`D-11`):** `opts` key `:max_rows` default `@default_max_rows`; query `limit: max_rows + 1`; if more than `max_rows` rows, set `truncated: true`, slice to `max_rows`, accurate `returned_count`.
5. **Return convention:** `{:ok, %{data: _, truncated: _, returned_count: _, max_rows: _}}` for success paths; DB failures follow Ecto (exceptions)—document in `@moduledoc` (`D-15`, aligns with Query + Ecto).
6. **Empty result:** `truncated: false`, `returned_count: 0`, CSV is header-only (or single header row, zero data rows—document one contract in `@moduledoc`); JSON `changes: []`.
7. **`count_matching(filters, opts) :: {:ok, %{count: non_neg_integer()}}`** — same `validate_timeline_filters!` + `export_changes_query` (or `timeline_query`) pipeline without row payload load; `Repo.aggregate(..., :count, :id)` on `AuditChange` binding for `--dry-run` in 14-02.

### Task 4: Integration tests — CSV + JSON happy path, filter parity, strict keys, empty, truncation, encoding edge

**What to do**

1. New `test/threadline/export_test.exs` using `Threadline.DataCase`, `Threadline.Test.Repo`, patterns from `test/threadline/query_test.exs` (`insert_transaction/1`, `insert_change/2`, `actor!/2` or equivalents).
2. **Happy path CSV:** Insert ≥2 changes with known `table_name` / `op`; `to_csv_iodata`; parse rows with NimbleCSV or split lines; assert column count and decode JSON cells with `Jason.decode!/1`.
3. **Happy path JSON:** `Jason.decode!(:binary.part(data, 0, byte_size(data)))` on result; assert `format_version == 1`, `is_binary(generated_at)`, length `changes` matches expected; each change has nested `transaction` with keys `id`, `occurred_at`, `actor_ref`, `source`.
4. **NDJSON:** `opts` with `json_format: :ndjson`; assert each non-empty line `Jason.decode!/1` succeeds and same keys as array mode.
5. **Filter parity:** For a fixed filter set, `Enum.map(Threadline.timeline(filters, opts), & &1.id)` equals `extracted_ids_from_export(...)` (same multiset; order both `captured_at` desc for deterministic asserts).
6. **Strict filters:** `assert_raise ArgumentError`, fn -> `Threadline.Export.to_csv_iodata([repo: Repo, oops: true], [])` end (adjust to actual API).
7. **Empty:** filters matching zero rows → `returned_count == 0`, `changes == []` / CSV header-only per Task 2 contract.
8. **Truncation:** Insert `max_rows + 2` changes with **distinct** `(captured_at, id)`; `max_rows: 3` → `truncated == true`, `returned_count == 3`.
9. **Encoding:** One row with `","`, newline, or UTF-8 in a text field—CSV round-trip without parse errors.
10. **Keyset / tie-break (optional but recommended):** two rows same `captured_at` (microsecond) if DB allows—assert stable ordering or document skip if PG rounds.
11. **`stream_changes/2`:** Insert ≥4 rows with distinct `(captured_at, id)`; `stream_changes(filters, [repo: Repo, page_size: 2]) |> Enum.to_list()` length equals 4 and ids match `timeline/2` order (proves multi-page pull without single unbounded `all`).
